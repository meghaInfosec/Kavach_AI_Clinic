## A.1 Source PCAP

**Source:** Malware-Traffic-Analysis.net, 2021 traffic analysis exercise
**Capture name:** 2021-05-26-Trickbot-infection-with-Cobalt-Strike.pcap
**SHA256:** fee3f9a285ab295821a7452a80cfaab25b8ccc60ad39052547ec67c8c3c9ca1e

- Contains C2 beaconing — periodic Trickbot check-ins to its C2 infrastructure,

- Meridian FinServe's environment — mid-sized financial firm with an internal AD
  domain, handling customer PII and payment data — matches the exact victim profile.
- The attack progression (workstation compromise → credential harvesting → lateral
  movement → data exfiltration) mirrors a realistic "patient zero" scenario.

**Limitations:**
- Capture comes from a generic exercise environment — host/domain names and IP ranges
  are not Meridian-specific.
- Workstream A will map observed hosts to equivalent Meridian roles (e.g.,
  "DESKTOP-OG16DGY" → branch workstation, "VICTORYPUNK-DC" → Meridian domain
  controller) rather than treating the capture as a literal Meridian log.

## Entry 7 — B.3 XSS Payload Encoding Bypass

**Date/Time:** 10 June 2026, ~04:00 PM
**Tool:** Claude (Sonnet 4.6)

**Prompt:**
```
For Project KAVACH Workstream B, I'm testing Reflected XSS on DVWA at security level
"medium". My basic payload <script>alert(1)</script> is being filtered. The filter is
stripping "<script>" using str_replace in PHP.

Without using any automated scanner, explain:
1. Why str_replace-based filtering fails as a security control (1-2 sentences max)
2. Give me 3 bypass payloads that work against a case-sensitive str_replace filter
3. Give me 1 bypass that works even if the filter is case-insensitive
4. For each payload, explain WHY it bypasses the specific filter mechanism
```

**Response:**
```
### Why str_replace Fails
str_replace performs a single-pass literal substitution — it cannot handle case
variation, double encoding, alternate event handlers, or non-script execution contexts.
It is a denylist, and denylists are always incomplete.

### Bypass 1 — Case variation: <SCRIPT>alert(1)</SCRIPT>
### Bypass 2 — Nested injection: <scr<script>ipt>alert(1)</scr</script>ipt>
### Bypass 3 — Event handler (no script tag): <img src=x onerror=alert(1)>
### Bypass 4 — beats case-insensitive filter: <body onload=alert(1)>
[Each with explanation of why it bypasses the specific filter]
```

**My notes:**
All four payloads verified against DVWA medium level. Bypass 1 (`<SCRIPT>`) confirmed immediately. Bypass 3 (`<img onerror>`) was the one used in the final finding report as most business-relevant — no script tag makes it harder to detect by basic WAF. The LLM explanation of *why* each bypass works was accurate and used directly in the Root Cause section of `2.XSS_Reflected/1.Reflected_XSS.md`. No errors in this response.
