## Entry 4 — B.3 XSS Payload Encoding Bypass

**Date/Time:** 10 June 2026, ~04:00 PM

**AI Tool Used:** Claude (Sonnet 4.6)

**Prompt:**
```
For Project KAVACH Workstream B, I'm testing Reflected XSS on DVWA at security level
"medium". My basic payload <script>alert(1)</script> is being filtered. The filter is
stripping "<script>" using str_replace in PHP.

Without using sqlmap or any automated scanner, explain:
1. Why str_replace-based filtering fails as a security control (1-2 sentences max)
2. Give me 3 bypass payloads that work against a case-sensitive str_replace('<script>', '', $input) filter
3. Give me 1 bypass that would also work if the filter was case-INSENSITIVE (i.e., strips all variants of <script>)
4. For each payload, explain WHY it bypasses the specific filter mechanism

This is for a DVWA lab (intentionally vulnerable app). Output as markdown.
```

**Response:**
```
## XSS Filter Bypass — str_replace Analysis

### Why str_replace Fails as a Security Control
`str_replace` performs a single-pass literal string substitution — it cannot account
for case variation, double encoding, alternate event handlers, or non-script HTML
execution contexts. It is a denylist, and denylists are always incomplete.

### Bypass 1 — Case Variation (beats case-sensitive filter)
Payload: `<SCRIPT>alert(1)</SCRIPT>`
Why it works: PHP's `str_replace('<script>', '', $input)` is case-sensitive by default.
`<SCRIPT>` does not match the literal `<script>` pattern, so it passes through unchanged.

### Bypass 2 — Nested Injection (beats case-sensitive filter)
Payload: `<scr<script>ipt>alert(1)</scr</script>ipt>`
Why it works: The filter removes `<script>` from the middle of the string, which
reconstructs the outer `<script>` tag in the remaining characters. Single-pass
substitution cannot handle this reconstruction.

### Bypass 3 — Event Handler (beats case-sensitive filter, no script tag)
Payload: `<img src=x onerror=alert(1)>`
Why it works: The filter only strips `<script>` tags — it has no rule for `<img>`
or other HTML elements with JavaScript event attributes. Any element with an `onerror`,
`onload`, or `onclick` handler executes JavaScript without a `<script>` tag at all.

### Bypass 4 — Event Handler, No Tag Name (beats case-insensitive filter)
Payload: `<body onload=alert(1)>`
Why it works: Even a case-insensitive filter that strips all `<script>` variants
cannot enumerate every HTML element. A `<body>` tag with `onload` triggers on page
load without any script tag — the filter has no coverage over this vector.
```

**My notes:**
All four payloads were verified against DVWA medium level. Bypass 1 (`<SCRIPT>`) confirmed working immediately. Bypass 3 (`<img onerror>`) confirmed working and was the one documented in the final finding report as the most business-relevant (no script tag = harder to detect by basic WAF). The nested injection (Bypass 2) also worked but produced ugly output — didn't use it in the report. The LLM explanation of *why* each bypass works was accurate and used directly in the Root Cause section of `2.XSS_Reflected/1.Reflected_XSS.md`.

## Entry 6 — B.4 Remediation — Parameterized Query Patch

**Date/Time:** 11 June 2026, ~03:30 PM

**AI Tool Used:** ChatGPT (first attempt — **WRONG**) → Claude (Sonnet 4.6) for final

**Prompt (used in both tools):**
```
For Project KAVACH B.4, I need a code-level patch for the SQL Injection finding in DVWA.

The vulnerable code (DVWA low.php line 7) is:
$query = "SELECT first_name, last_name FROM users WHERE user_id = '$id';";
$result = mysqli_query($GLOBALS["___mysqli_ston"], $query);

Write a patch that:
1. Uses parameterized queries / prepared statements
2. Stays within the DVWA PHP + MySQLi environment (don't switch to PDO unless necessary)
3. Shows as a proper code diff (before/after)
4. Includes a 1-sentence explanation of WHY parameterized queries prevent injection
   that a non-PHP developer could understand

Also: what is the ONE alternative fix the team considered and rejected, and what does
that alternative cost or break?
```

**Response from ChatGPT (first attempt — WRONG):**
```
Here's a parameterized query fix for DVWA:

$stmt = $conn->prepare("SELECT first_name, last_name FROM users WHERE user_id = ?");
$stmt->bind_param("i", $id);   // ← "i" for integer
$stmt->execute();
$result = $stmt->get_result();
```

**Why ChatGPT's response was wrong:**
The patch used `bind_param("i", $id)` with type `"i"` (integer). However, in DVWA's
actual implementation, `$id` is passed as a string (from `$_REQUEST['id']`), and
MySQLi's `bind_param` with `"i"` will silently truncate a string like `1' OR '1'='1`
to `1` — this appears to work, but the type binding behaviour is implementation-
dependent. More importantly, the DVWA `low.php` uses `$GLOBALS["___mysqli_ston"]`
as the connection object — the patch used `$conn` which is not defined in that scope,
so it would produce a fatal PHP error when run. We caught this by actually running the
patched code and getting a `Fatal error: Call to a member function prepare() on null`.

**Response from Claude (final version — correct):**
```diff
- $query  = "SELECT first_name, last_name FROM users WHERE user_id = '$id';";
- $result = mysqli_query($GLOBALS["___mysqli_ston"],  $query );

+ $stmt = mysqli_prepare($GLOBALS["___mysqli_ston"],
+     "SELECT first_name, last_name FROM users WHERE user_id = ?");
+ mysqli_stmt_bind_param($stmt, "s", $id);   // "s" = string, matches $_REQUEST type
+ mysqli_stmt_execute($stmt);
+ $result = mysqli_stmt_get_result($stmt);
```

**Why it works:** Parameterized queries separate the SQL instruction from the data at
the protocol level — the database driver sends them in separate channels, so the
database engine never interprets `$id`'s content as SQL syntax regardless of what
it contains.

**Alternative considered and rejected:**
`mysqli_real_escape_string($GLOBALS["___mysqli_ston"], $id)` — rejected because
escaping is encoding-dependent and has been bypassed under specific character-set
configurations (e.g., GBK multi-byte). Parameterization separates instruction from
data at the protocol level; escaping only transforms the data. A parameterized
query is stronger and has no known bypass class.
```

**My notes:**
This is the most important failure mode in the entire engagement's LLM log. ChatGPT confidently produced a patch that looked correct and would pass a code review by eye — but broke at runtime due to a wrong variable name (`$conn` vs `$GLOBALS["___mysqli_ston"]`). We caught it only because we actually ran the patched code. The lesson: LLM code patches must be run, not just read. Claude's version used the correct procedural MySQLi API that matches the existing DVWA codebase and ran without error.

```

**Response:**
```
## Semgrep CE — SAST Baseline Commands

### Before Scan (baseline — run BEFORE any patches)
```bash
semgrep --config=p/owasp-top-ten \
        --json \
        --output=before.json \
        ./dvwa/ ./juice-shop/
```

### After Scan (post-patch — run AFTER all fixes are applied)
```bash
semgrep --config=p/owasp-top-ten \
        --json \
        --output=after.json \
        ./dvwa/ ./juice-shop/
```

### Key JSON Fields
- `id`: the rule that fired (e.g., `php.lang.security.mysqli-concat`)
- `path`: file path where the pattern was found
- `start/end`: line and column range of the flagged code
- `extra.message`: human-readable description of why this is flagged
- `extra.severity`: ERROR (critical/high) or WARNING (medium/low)

### Interpreting after.json
- **Zero results** (`"results": []`): all flagged patterns removed. If after patching
  you expect zero findings and get zero, this is valid — document it as "all patched".
  Do NOT leave after.json empty without explanation; a reviewer will assume the scan
  was not run.
- **Residual finding**: if a finding remains because a compensating control was applied
  rather than the root pattern removed (e.g., auth gate on a static-serve route),
  Semgrep will still flag the syntax. Keep the finding in after.json and add an inline
  `NOTE:` comment in the `extra.message` field explaining the compensating control.
  This makes the residual intentional and documented rather than appearing as an
  oversight.
```

**My notes:**
The `--config=p/owasp-top-ten` flag was the key piece we were missing — we had been running Semgrep with `--config=auto` which pulls in too many rules and creates noise. Switching to `p/owasp-top-ten` produced exactly the 5 findings that matched our manually found vulnerabilities. The note about documenting residual findings in `after.json` directly shaped how we handled the `directory-listing` residual in `B.4 SAST/2.after.json`.
