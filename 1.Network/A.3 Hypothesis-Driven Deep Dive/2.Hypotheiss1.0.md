## Executive Summary

This report documents the forensic analysis of a network packet capture (PCAP) containing a real-world **Trickbot malware infection** that led to **Cobalt Strike deployment**, **Active Directory compromise**, and **data exfiltration**. Using Zeek for log generation and RITA for automated behavioral analysis, the analyst identified the full attack chain — from initial infection of the Domain Controller through lateral movement to six additional hosts.

**Severity: CRITICAL**

---

## Hypothesis

Each hypothesis below follows a structured three-part format:

1. **Hypothesis Statement** — what we suspect before looking at evidence
2. **Points that would CONFIRM it** — what we expect to see in logs if true
3. **Points that would REFUTE it** — what we expect to see in logs if false
4. **Verdict** — what the actual evidence shows

---

### H1 — The Domain Controller (10.5.26.4) is the Initial Infection Point

> *"The Domain Controller `10.5.26.4` is the first host infected by Trickbot and is actively beaconing to external C2 infrastructure using fake or typosquatted domains over HTTP."*

#### Points That Would CONFIRM This
- `10.5.26.4` initiates outbound connections to external IPs before any other internal host does
- Connections to suspicious or typosquatted domains (e.g. fake antivirus names) originating from `10.5.26.4`
- HTTP POST requests carrying data out to external IPs from `10.5.26.4`
- Spoofed or anomalous User-Agent strings in HTTP traffic from `10.5.26.4`
- RITA flags `10.5.26.4` as a beaconing source with high beacon score
- `10.5.26.4` appears in conn.log as the first internal host with repeated external connections

#### Points That Would REFUTE This
- Another internal host (e.g. `10.5.26.132`) initiates outbound C2 connections earlier than `10.5.26.4`
- `10.5.26.4` only communicates with known legitimate IPs (Microsoft, Windows Update, DNS)
- No POST requests or data uploads from `10.5.26.4` to external hosts
- All outbound connections from `10.5.26.4` have normal durations and known destinations
- RITA shows no beacon activity from `10.5.26.4`

#### Verdict — ✅ CONFIRMED

Evidence from `conn.log` and `http.log` shows `10.5.26.4` making repeated connections to `5.199.162.3:80` with highly consistent short durations (0.51s–0.60s), one sustained connection of 101.3s, and multiple POST requests to `antivirusupdaty.com/as` — a typosquatted fake antivirus domain. RITA rated this as HIGH severity with a 60% beacon score. A spoofed Android Pixel C User-Agent was detected on a Domain Controller — a clear anomaly. No other internal host initiated outbound C2 before `10.5.26.4`.

### H2 — Cobalt Strike Was Deployed as a Second-Stage Payload After Trickbot

> *"Trickbot downloaded and executed Cobalt Strike as a post-exploitation framework. Cobalt Strike beacons will communicate to C2 servers using structured URIs containing hostname and unique beacon IDs, over HTTPS to multiple redundant servers."*

#### Points That Would CONFIRM This
- HTTP URIs containing hostname, OS build string, and a hex beacon ID in a structured format
- Multiple external C2 IPs receiving identical URI patterns (redundant C2 channels)
- Two different internal hosts beaconing to the same external Cobalt Strike C2 IP
- High RITA beacon scores (>70%) to external IPs on port 443
- Long sustained connection durations (>30 minutes) indicating active C2 sessions
- Random long-named `.exe` files dropped via SMB consistent with Cobalt Strike stager

#### Points That Would REFUTE This
- HTTP URIs are generic or follow no structured naming convention
- Only one external C2 IP observed — no redundancy
- All HTTPS connections go to known legitimate services (CDNs, Microsoft, Google)
- No EXE files dropped via SMB
- RITA shows no beacon activity to port 443 external IPs
- Only `10.5.26.4` is beaconing, no spread to other hosts

#### Verdict — ✅ CONFIRMED

`http.log` revealed Cobalt Strike malleable C2 profile URIs with embedded hostname, OS build, and unique beacon IDs:
```
POST /rob87/DESKTOP-OG16DGY_W617601.D5D933B2E59558F3F46D9130BB091D98/90
POST /tot108/VICTORYPUNK-DC_W617601.B5EFB19D6BB313B551F3FB5897A283BE/90
```
Both `10.5.26.4` and `10.5.26.132` were beaconing to the same C2 IP `5.202.120.150:443`, confirming spread. RITA scored `10.5.26.132 → 37.228.70.134` at 87.1% over 42 minutes and `10.5.26.4 → 45.229.71.211` at 75% over 46 minutes. Two random-named EXEs were dropped via SMB to the DC's `\WINDOWS\` directory.

### H3 — Lateral Movement Occurred via SMB and Remote Service Execution

> *"Once Cobalt Strike was active, the attacker moved laterally using SMB to drop executables and remotely created Windows services (PsExec-style) to execute them on target hosts."*

#### Points That Would CONFIRM This
- SMB file write operations from a compromised host to another internal host
- Random or unusually long executable filenames written to system directories like `\WINDOWS\`
- DCE-RPC `svcctl` calls in sequence: `OpenSCManagerW → CreateServiceW → StartServiceW → DeleteService`
- `DeleteService` called immediately after `StartServiceW` (anti-forensics cleanup)
- SMB connections to admin shares or named pipes like `\svcctl`, `lsarpc`, `samr`
- Multiple internal hosts all connecting to the DC in a short time window

#### Points That Would REFUTE This
- No SMB file write operations between internal hosts
- No `svcctl` RPC calls observed in `dce_rpc.log`
- Only legitimate GPO-related SMB traffic (gpt.ini reads) from workstations
- No executable files transferred via SMB
- All internal host-to-host traffic is normal (DNS, Kerberos, LDAP only)

#### Verdict — ✅ CONFIRMED

`smb_files.log` shows `10.5.26.132` writing two randomly-named executables to `10.5.26.4\WINDOWS\` at timestamp `1622061083`. `dce_rpc.log` shows the complete PsExec chain immediately after:
```
svcctl → OpenSCManagerW → OpenServiceW → CreateServiceW → StartServiceW → DeleteService → CloseServiceHandle
```
The `DeleteService` call confirms deliberate anti-forensics. Named pipes `\svcctl`, `lsarpc`, `samr`, and `netlogon` were all accessed, consistent with Cobalt Strike's `jump psexec` module.

### H4 — DNS Tunneling Was Used as a Covert C2 Channel

> *"Trickbot abused DNS to tunnel C2 traffic, either through high-entropy subdomain queries to a specific domain or through regular high-frequency connections to public DNS resolvers like 8.8.8.8."*

#### Points That Would CONFIRM This
- High beacon score from RITA on connections from internal hosts to `8.8.8.8` or `1.1.1.1`
- Abnormally large `dns.log` file size relative to the network activity
- High frequency of DNS queries with regular timing intervals
- DNS query responses carrying unusually large payloads
- Connections to public resolvers that show beaconing patterns rather than random lookup patterns

#### Points That Would REFUTE This
- `dns.log` contains only normal domain lookups (Microsoft, Windows Update, browser traffic)
- No beaconing pattern observed in connections to DNS resolvers
- RITA shows no anomaly in DNS traffic
- DNS query volume is proportional and irregular (consistent with user-driven browsing)
- All DNS queries resolve to known legitimate domains

#### Verdict — ✅ CONFIRMED

RITA detected an 87.6% beacon score on `10.5.26.4 → 8.8.8.8` — the highest beacon score in the dataset. A Domain Controller does not legitimately beacon to Google's public DNS resolver at machine-like regularity. The `dns.log` was 101 KB in size — disproportionately large compared to the overall traffic volume — indicating high DNS query volume consistent with data tunneling. Legitimate DNS resolution produces irregular, user-driven patterns, not a near-perfect beacon score.

### H5 — Harvested Credentials Were Abused via DCSync and Kerberos

> *"After gaining access to the Domain Controller, the attacker performed a DCSync attack to dump all Active Directory password hashes, then used the stolen credentials to request Kerberos tickets for further lateral movement."*

#### Points That Would CONFIRM This
- `drsuapi` RPC calls (`DRSBind`, `DRSCrackNames`, `DRSUnbind`) from non-DC workstations to the DC
- Multiple workstations performing the same `drsuapi` sequence (indicating scripted/automated execution)
- Kerberos AS-REQ failures followed immediately by AS-REP success (password spray or pass-the-hash)
- TGS ticket requests for sensitive services (`cifs`, `ldap`, `host`) from a compromised user account
- User account requesting tickets for services on machines other than their own workstation

#### Points That Would REFUTE This
- `drsuapi` calls only originate from `10.5.26.4` (the DC itself — which is normal)
- No workstations call `drsuapi` against the DC
- Kerberos log shows only clean AS-REQ/TGS flows with no failures
- No user accounts request tickets for unusual services or foreign machines
- Only machine accounts (`DESKTOP-*$`) perform Kerberos operations, no human user accounts

#### Verdict — ✅ CONFIRMED

`dce_rpc.log` shows all six compromised workstations (`10.5.26.130`, `.132`, `.134`, `.136`, `.138`, `.140`) calling `drsuapi DRSBind → DRSCrackNames → DRSUnbind` against the DC — a pattern that is never legitimate from workstations. `kerberos.log` shows `rebecca.vargas` authenticating from `10.5.26.132` with an initial AS-REQ failure followed by success, then requesting TGS tickets for `cifs`, `ldap`, and `host` services — confirming harvested credential reuse for lateral movement across the domain.

## Finding 1 — C2 Beaconing

### 1.1 Trickbot HTTP Beaconing

**Source Log:** `conn.log`, `http.log`

The Domain Controller `10.5.26.4` made repeated short-duration TCP connections to `5.199.162.3` on port 80 with highly consistent timing — a hallmark of automated malware beaconing.

```
10.5.26.4 → 5.199.162.3:80   duration: 0.51s
10.5.26.4 → 5.199.162.3:80   duration: 0.52s
10.5.26.4 → 5.199.162.3:80   duration: 0.56s
10.5.26.4 → 5.199.162.3:80   duration: 0.60s
10.5.26.4 → 5.199.162.3:80   duration: 101.3s  ← data transfer/tasking
```

The consistent sub-second durations indicate polling, while the single long-duration connection represents active command retrieval or tasking.

**RITA Score:** HIGH — 60% beacon score to `antivirusupdaty.com`

**Rare User-Agent detected by RITA:**
```
Mozilla/5.0 (Linux; Android 7.0; Pixel C Build/NRD90M; wv) AppleWebKit/537.36
```
A Domain Controller impersonating an Android mobile device is a clear indicator of a spoofed User-Agent, consistent with Trickbot's evasion techniques.

### 1.2 Cobalt Strike HTTPS Beaconing

**Source Log:** `conn.log`, `http.log`, RITA

Two internal hosts were found beaconing to the same external Cobalt Strike C2 IP (`5.202.120.150:443`), confirming the beacon had spread to a second machine.

```
10.5.26.4   → 5.202.120.150:443   (C2 session)
10.5.26.132 → 5.202.120.150:443   (C2 session — spread confirmed)
```

Cobalt Strike C2 URIs with embedded hostname and beacon ID were observed in `http.log`:

```

POST /tot108/VICTORYPUNK-DC_W617601.B5EFB19D6BB313B551F3FB5897A283BE/90
```

URI structure breakdown:
- `/rob87/` — Campaign ID
- `DESKTOP-OG16DGY_W617601` — Hostname + OS build
- `D5D933B2E59558F3F46D9130BB091D98` — Unique beacon ID
- `/90` — Jitter value

**RITA Scores:**

- `10.5.26.4 → 45.229.71.211` — **75.00% beacon**, 46m0s duration

## Finding 3 — Lateral Movement

### 3.1 Remote Executable Deployment via SMB (PsExec-style)

**Source Log:** `smb_files.log`

Two randomly-named executables were written to the `\WINDOWS\` directory of the Domain Controller from `10.5.26.132` via SMB:

```
Timestamp: 1622061083
Source:    10.5.26.132
Target:    10.5.26.4
Action:    SMB::FILE_OPEN
Files:
  \WINDOWS\h48l10jxplwhq9eowyecjmwg0nxwu72zblns1l3v3c6uu6p6069r4c4c5yjwv_e7.exe
  \WINDOWS\rt59qhcmr9hva1x_lvqf22wukxhl648rs7rzjcx5x6f8u1mx7yeh1xwbn3muqja2.exe
```

Random long filenames are a signature of Cobalt Strike's `jump psexec` lateral movement technique, designed to evade static detection rules.

---

### 3.2 Remote Service Creation and Deletion via DCE-RPC

**Source Log:** `dce_rpc.log`

Following the file drop, `10.5.26.132` used the Windows Service Control Manager (svcctl) remotely to create, start, and delete a service on the DC — the classic PsExec execution chain:

```
10.5.26.132 → 10.5.26.4   svcctl   OpenSCManagerW      ← open service manager
10.5.26.132 → 10.5.26.4   svcctl   OpenServiceW        ← check if exists
10.5.26.132 → 10.5.26.4   svcctl   CreateServiceW      ← install service
10.5.26.132 → 10.5.26.4   svcctl   StartServiceW       ← execute payload
10.5.26.132 → 10.5.26.4   svcctl   DeleteService       ← clean up evidence
10.5.26.132 → 10.5.26.4   svcctl   CloseServiceHandle  ← close connection
```

The `DeleteService` call immediately after execution is a deliberate anti-forensics step to remove evidence of the service from the registry.

---

### 3.3 DCSync Attack — Active Directory Credential Dump

**Source Log:** `dce_rpc.log`

The `drsuapi` (Directory Replication Service) RPC interface is used exclusively by Domain Controllers for AD replication. When called from non-DC workstations, it indicates a **DCSync attack** — simulating a DC to pull password hashes for all domain accounts without needing local access to the DC.

All six compromised workstations performed DCSync against the DC:

```
10.5.26.132 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
10.5.26.130 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
10.5.26.134 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
10.5.26.136 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
10.5.26.138 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
10.5.26.140 → 10.5.26.4   drsuapi   DRSBind → DRSCrackNames → DRSUnbind
```

This indicates Cobalt Strike's `dcsync` command was executed on all compromised hosts, effectively giving the attacker NTLM hashes for every domain account including `krbtgt` — enabling **Golden Ticket** creation.

---

### 3.4 Kerberos Credential Abuse

**Source Log:** `kerberos.log`

After DCSync, the attacker used harvested credentials (`rebecca.vargas`) to authenticate and request Kerberos service tickets for lateral movement:

```
10.5.26.132 → 10.5.26.4   AS-REQ   rebecca.vargas/VICTORYPUNK   → FAILED (F)
10.5.26.132 → 10.5.26.4   AS-REP   rebecca.vargas/VICTORYPUNK   → SUCCESS (T)
10.5.26.132 → 10.5.26.4   TGS      rebecca.vargas → cifs/VictoryPunk-DC     (T)
10.5.26.132 → 10.5.26.4   TGS      rebecca.vargas → ldap/VictoryPunk-DC     (T)
10.5.26.132 → 10.5.26.4   TGS      rebecca.vargas → host/desktop-og16dgy    (T)
```

The initial AS-REQ failure followed by success is consistent with a **password spray or pass-the-hash** attempt. TGS tickets for `cifs`, `ldap`, and `host` services indicate preparation for file access, directory enumeration, and remote execution.

## Conclusion & Recommendations

### Conclusion

This investigation confirms a complete, multi-stage attack chain by a sophisticated threat actor using **Trickbot** as an initial access broker and **Cobalt Strike** as the post-exploitation framework. The attacker achieved full domain compromise including credential dumping of all Active Directory accounts via DCSync across all six workstations.

**All five hypotheses were confirmed by forensic evidence.**

### Immediate Response Actions

2. **Reset all domain credentials** — especially `krbtgt` (twice, 10 hours apart) to invalidate Golden Tickets
3. **Reset `rebecca.vargas`** and all other domain user passwords
4. **Block all malicious IPs and domains** at the perimeter firewall
5. **Preserve memory dumps** from all affected hosts before remediation

### Detection Recommendations

1. **Alert on `drsuapi` calls from non-DC hosts** — DCSync is nearly always malicious
2. **Alert on `svcctl CreateService + DeleteService` in sequence** — PsExec cleanup pattern
3. **Monitor for random long-filename EXEs written via SMB** to sensitive directories
4. **Deploy RITA continuously** against live Zeek logs for automated beacon scoring
5. **Alert on high-frequency connections to public DNS resolvers** (8.8.8.8, 1.1.1.1) — DNS tunneling indicator
6. **Block outbound HTTP POST requests** to newly-registered or uncategorized domains

### Hardening Recommendations

1. Enable **Protected Users Security Group** for privileged accounts
2. Implement **Privileged Access Workstations (PAWs)** for domain admin activity
3. Deploy **Microsoft Defender Credential Guard** to prevent credential theft
4. Enable **SMB signing** to prevent relay attacks
5. Implement **network segmentation** — workstations should not communicate directly with each other
6. Deploy **Zeek + RITA** in continuous monitoring mode on all network segments
